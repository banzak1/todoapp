Sessão de continuidade da Fase 8 (Observabilidade), com isolamento da feature em uma

`observability-foundation`.

## O que foi entregue

### Diagrama de arquitetura

- Diagrama versionado como imagem JPEG, HTML interativo e JSON-fonte em `notes/`.
- README atualizado para exibir a arquitetura do sistema.
- Commit: `a397a36 docs: add system architecture diagram`.

### T2 — Health checks e exposição segura do Actuator

- Actuator configurado com `/actuator/health/liveness` e
  `/actuator/health/readiness`.
- Liveness depende apenas do estado interno da aplicação.
- Readiness depende do estado interno e do banco; Kafka não é gate de readiness.
- Health sanitizado, sem detalhes internos de componentes.
- Produção expõe somente `health`, `info` e `prometheus`.
- `KafkaAdmin` não cria tópicos automaticamente no startup
  (`spring.kafka.admin.auto-create=false`).
- Handler global corrigido para retornar `404` a recursos não expostos, como
  `/actuator/env`.
- README contém comandos de validação local com H2 e `curl`.
- Teste: `HealthEndpointTest` passou com H2.
- Commit: `2da7307 feat(observability): configure actuator health probes`.

### T3 — Correlation ID HTTP

- Novo filtro HTTP `CorrelationIdFilter`.
- Aceita `X-Correlation-ID` UUID canônico válido ou gera outro UUID seguro.
- Devolve o ID no header da resposta e o mantém no MDC como `correlationId` durante a
  requisição.
- O MDC é limpo em `finally`, evitando vazamento entre threads reutilizadas.
- Headers vazios, inválidos ou excessivamente longos são substituídos.
- Validação manual feita com endpoint de health; POST depende do broker Kafka para
  concluir o fluxo de evento.
- Teste: `CorrelationIdFilterTest` passou com 3 cenários.
- Commit: `5affa21 feat(observability): add HTTP correlation ID filter`.

## Decisões registradas

- DEC-032: liveness e readiness possuem dependências distintas.
- DEC-033: tópicos Kafka são responsabilidade da infraestrutura, não da aplicação.
- DEC-034: correlation ID explícito no HTTP usa UUID canônico e MDC temporário.
